title: GitHub
description: Użyj legion w problemach z GitHubem i zastosujch ściągnięcia.
---

legion integruje się z przepływem pracy w GitHub. Wspomnij o `/legion` lub `/oc` w swoim komentarzu, a legion wykonaj zadania w ramach modułu uruchamiającego GitHub Actions.

---

## Cechy

- **Problem związany z segregacją**: Poproś legion o szczegółowe wyjaśnienie problemu i wyjaśnienie go.
- **Napraw i zaimplementuj**: Poproś legion o naprawienie problemu lub zaimplementowanie funkcji. Będzie dostępny w następnym oddziale i wysyłać PR ze stosowaniem dodatku.
- **Bezpieczny**: legion działa w modułach sprzętowych GitHuba.

---

## Instalacja

Uruchomione dalsze postępowanie w przypadku wystąpienia w repozytorium GitHub:

```bash
legion github install
```

Aby przeprowadzić Cię przez proces instalacji aplikacji GitHub, utwórz działanie i skonfiguruj wpisy tajnych.

---

### Konfiguracja ręczna

Można też uszkodzić to rozwiązanie.

1. **Zainstaluj aplikację GitHub**

   Wejdź na [**github.com/apps/legion-agent**](https://github.com/apps/legion-agent). wystąpienie się, że jest natychmiastowe w repozytorium usuwam.

2. **Dodaj przepływ pracy**

   Dodaj zdalny plik pracy do `.github/workflows/legion.yml` w swoim repozytorium. wystąpił, że ustawiłeś sędziego `model` i wymagany klucz API w `env`.

   ```yml title=".github/workflows/legion.yml" {24,26}
   name: legion

   on:
     issue_comment:
       types: [created]
     pull_request_review_comment:
       types: [created]

   jobs:
     legion:
       if: |
         contains(github.event.comment.body, '/oc') ||
         contains(github.event.comment.body, '/legion')
       runs-on: ubuntu-latest
       permissions:
         id-token: write
       steps:
          - name: Checkout repository
            uses: actions/checkout@v6
            with:
              fetch-depth: 1
              persist-credentials: false

          - name: Run Legion
           uses: wearethelegion/legion/github@latest
           env:
             ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
           with:
             model: anthropic/claude-sonnet-4-20250514
             # share: true
             # github_token: xxxx
   ```

3. **Przechowuj klucze API w tajemnicy**

   W **ustawieniach** swojej organizacji lub projektu rozwiń **Sekretne i zmienne** po lewej stronie i wybierz **Działania**. Dodałem wymagane klucze API.

## Konfiguracja

- `model`: Model używany z legion. Przyjmuje format `provider/model`. To **wymagane**.
- `agent`: Agent, którego należy używać. Musi być odległym agentem. Wraca do `default_agent` z konfiguracją lub `"build"`, jeśli nie został znaleziony.
- `share`: Czyć funkcję sesji legion. Domyślnie **true** dla repozytoriów publicznych.
- `prompt`: Opcjonalny niestandardowy monit o zastąpienie przestrzegania zachowania. Wykorzystanie tego, aby dostosować sposób przetwarzania przez legion.
- `token`: opcjonalny token dostępu GitHub podstawowe operacje, takie jak tworzenie komentarzy, zatwierdzanie zmian i otwieranie zastosowania ściągnięcia. Domyślnie legion używa tokena dostępu do instalacji z aplikacji legion GitHub, więc zatwierdzenia, komentarze i zasady ściągnięcia widoczne jako źródło z aplikacji.

  Alternatywnie możesz użyć [wbudowanego `GITHUB_TOKEN`](https://docs.github.com/en/actions/tutorials/authenticate-with-github_token) modułu uruchamiającego GitHub Action bez instalacji aplikacji legion GitHub. Pamiętaj tylko o przyznaniu wymaganych mocy w przepływie pracy:

  ```yaml
  permissions:
    id-token: write
    contents: write
    pull-requests: write
    issues: write
  ```

  Jeśli chcesz, możesz także użyć [osobistych tokenów dostępu](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens)(PAT).

## Obsługiwane wydarzenia

legion może zostać wywołany przez zdarzenie GitHub:

| Typ zdarzenia                 | Wywołane przez                                   | Szczegóły                                                                                                                                             |
| ----------------------------- | ------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `issue_comment`               | Skomentuj problem lub PR                         | Wspomnij o `/legion` lub `/oc` w swoim komentarzu. legion odczytuje kontekst i może być częścią składową, otwieraną przez PR lub odpowiedzialną.  |
| `pull_request_review_comment` | Komentarz określonym kodem w PR                  | Wspomnij o `/legion` lub `/oc` podczas przeglądania kodu. legion źródło pochodzenia, numery linii i kontekst różnicowy.                           |
| `issues`                      | Wydanie otwarte lub edytowane                    | Automatycznie wyzwalaj legion po utworzeniu lub zmodyfikowaniu problemów. Wymaga wejścia `prompt`.                                                  |
| `pull_request`                | PR otwarty lub zaktualizowany                    | Automatycznie wyzwalaj legion, gdy PR są otwierane, synchronizowane lub ponownie otwierane. Przydatne w przypadku automatycznych znajomych.         |
| `schedule`                    | Harmonogram oparty na Cron                       | Uruchom legion zgodnie z harmonogramem. Wymagane wejście `prompt`. Dane wejściowe trafiają do dzienników i trafiań PR (nie ma problemu z recenzją). |
| `workflow_dispatch`           | Ręczny wyłącznik z interfejsu użytkownika GitHub | Uruchom legion na karcie Akcje. Wymagane wejście `prompt`. Dane wejściowe trafiają do dzienników i odbiorców PR.                                    |

### Przykład harmonogramu

Uruchamiaj legion zgodnie z harmonogramem, aby wykonać zautomatyzowane zadania:

```yaml title=".github/workflows/legion-scheduled.yml"
name: Scheduled Legion Task

on:
  schedule:
    - cron: "0 9 * * 1" # Every Monday at 9am UTC

jobs:
  legion:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: write
      pull-requests: write
      issues: write
    steps:
      - name: Checkout repository
        uses: actions/checkout@v6
        with:
          persist-credentials: false

      - name: Run Legion
        uses: wearethelegion/legion/github@latest
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        with:
          model: anthropic/claude-sonnet-4-20250514
          prompt: |
            Review the codebase for any TODO comments and create a summary.
            If you find issues worth addressing, open an issue to track them.
```

W przypadku wystąpienia danych wyjściowych `prompt` są **wymagane**, które nie są dostarczane, z których można wyodrębnić instrukcje. Zaplanowane przepływomierze pracy bez kontekstu użytkownika, który został uruchomiony, więc przepływ pracy musi `contents: write` i `pull-requests: write`, wystąpisz, że legion utworzył główne lub PR.

### Przykład żądania ściągnięcia

Automatycznie przeglądaj żądania ściągnięcia po ich otwarciu lub aktualizacji:

```yaml title=".github/workflows/legion-review.yml"
name: legion-review

on:
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
      pull-requests: read
      issues: read
    steps:
      - uses: actions/checkout@v6
        with:
          persist-credentials: false
      - uses: wearethelegion/legion/github@latest
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          model: anthropic/claude-sonnet-4-20250514
          use_github_token: true
          prompt: |
            Review this pull request:
            - Check for code quality issues
            - Look for potential bugs
            - Suggest improvements
```

W przypadku zdarzenia `pull_request`, jeśli nie podano `prompt`, legion użytkownika przeglądającego ściągnięcie.

### Przykład segregacji problemów

Automatycznie segreguj nowe problemy. Dziesięć przykładów filtruje do kont starszych niż 30 dni w celu ograniczenia spamu:

```yaml title=".github/workflows/legion-triage.yml"
name: Issue Triage

on:
  issues:
    types: [opened]

jobs:
  triage:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: write
      pull-requests: write
      issues: write
    steps:
      - name: Check account age
        id: check
        uses: actions/github-script@v7
        with:
          script: |
            const user = await github.rest.users.getByUsername({
              username: context.payload.issue.user.login
            });
            const created = new Date(user.data.created_at);
            const days = (Date.now() - created) / (1000 * 60 * 60 * 24);
            return days >= 30;
          result-encoding: string

      - uses: actions/checkout@v6
        if: steps.check.outputs.result == 'true'
        with:
          persist-credentials: false

      - uses: wearethelegion/legion/github@latest
        if: steps.check.outputs.result == 'true'
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        with:
          model: anthropic/claude-sonnet-4-20250514
          prompt: |
            Review this issue. If there's a clear fix or relevant docs:
            - Provide documentation links
            - Add error handling guidance for code examples
            Otherwise, do not comment.
```

W przypadku zdarzenia `issues` wprowadzenie `prompt` jest **wymagane**, ponieważ nie ma komentarza, z którego można wyodrębnić instrukcje.

## Niestandardowe monity

Zastąp domyślne monit, aby zastosować zachowanie legion do twojego własnego pracy.

```yaml title=".github/workflows/legion.yml"
- uses: wearethelegion/legion/github@latest
  with:
    model: anthropic/claude-sonnet-4-5
    prompt: |
      Review this pull request:
      - Check for code quality issues
      - Look for potential bugs
      - Suggest improvements
```

Jest to przepis wykonawczy, który podlega przepisom kodowania lub zakresów tematycznych dla twojego projektu.

## Przykłady

Oto kilka możliwości wykorzystania legion w GitHubie.

- **Wyjaśnij problem**

  Dodaj dziesięć komentarzy w numerze GitHub.

  ```
  /legion explain this issue
  ```

  legion przeczytaj całość, zawierając dodatek z komentarzami i odpowiedzią z jasnym wyjaśnieniem.

- **Napraw problem**

  W numerze GitHub powiedz:

  ```
  /legion fix this
  ```

  A legion utworzy nową podstawę, wdroży zmiany i zastąpi PR ze zmiany.

- **Przegląd zmiany PR i wprowadzenie zmiany**

  Zostaw komentarz w PR GitHub.

  ```
  Delete the attachment from S3 when the note is removed /oc
  ```

  legion zaimplementuje uruchomioną zmianę i zatwierdzi ją do tego samego PR.

- **Przejrzyj konkretne linie kodu**

  Zostaw komentarz bezpośrednio w wierszu kodu w dodatku „Pliki” PR. legion automatyczne wykrywanie pliku, numery linii i kontekst różnicowy, aby sprawdzić odpowiedzi.

  ```
  [Comment on specific lines in Files tab]
  /oc add error handling here
  ```

  Komentując określone linie, legion otrzymuje:
  - Dokładny plik, który jest sprawdzany
  - Konkretne linie kodu
  - Otaczający kontekst różnicowy
  - Informacje o numerze linii

  Dostępne na bardziej szczegółowe rozwiązanie bez konieczności stosowania ręcznego określania plików lub numerów wierszy.
